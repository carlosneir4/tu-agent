# tu-agent Claude Code plugin

Runs the tu-agent knowledge pipeline inside Claude Code, on your Claude
subscription â€” no API key needed. The tu-agent binary does everything
deterministic (graph build, concept clustering, card rendering, CLAUDE.md
registration); Claude Code does everything generative (concept definitions,
architecture overview).

## Requirements

- The `tu-agent` binary, version 0.3.0+, on your `PATH` or at
  `~/.tu-agent/bin/tu-agent`:

  ```bash
  go install -tags sqlite_fts5 github.com/carlosneir4/tu-agent/cmd/tu-agent@latest
  ```

## Install

From this repository:

```bash
claude --plugin-dir /path/to/tu-agent/plugin
```

## Graph MCP server

Installing this plugin registers a stdio MCP server, `tu-agent-graph`, that
Claude Code starts automatically. It runs the bundled binary
(`${CLAUDE_PLUGIN_ROOT}/bin/tu-agent mcp`) against the project you opened Claude
Code in, exposing graph query tools and test-generation helpers:

| Tool | Surfaces as | What it answers |
|------|-------------|-----------------|
| `get_impact` | `mcp__tu-agent-graph__get_impact` | Blast radius: what breaks if I change this symbol? |
| `get_context` | `mcp__tu-agent-graph__get_context` | Everything relevant to touching a target: blast radius, skills, conventions, tests. |
| `find_symbol` | `mcp__tu-agent-graph__find_symbol` | Where is this symbol defined? |
| `test_gaps` | `mcp__tu-agent-graph__test_gaps` | Rank untested public functions by risk (fan-in Ă— blast radius). |
| `test_scaffold` | `mcp__tu-agent-graph__test_scaffold` | Deterministic half of test generation: context, test path, scoped run command. |
| `get_concept` | `mcp__tu-agent-graph__get_concept` | Return a concept card (SKILL.md) by name, or list all concepts. |
| `get_traits` | `mcp__tu-agent-graph__get_traits` | Trait view: shared interfaces + logic sites (as_type) and implementers + blast radius (as_interface). JSON. |
| `get_flow` | `mcp__tu-agent-graph__get_flow` | Execution call tree from an entry symbol: boundary crossings, interface dispatch candidates, depth + fan-out control. JSON. |

The tools read `./.tu-agent/graph.db`. Build it first with `/tu-agent:learn`
(or `tu-agent graph build`); until a graph exists the queries return empty
results. The server refreshes the graph from the working tree each time it
starts.

## Dev-flow agents

`/tu-agent:prepare` sets up the repo end-to-end:

1. Runs the binary's deterministic setup â€” a `CLAUDE.md` knowledge block, a
   hardened `.claude/settings.json` (deny-wins permissions, a secret-guard hook,
   a toolchain formatter hook, and an `enabledMcpjsonServers` allowlist), a
   private-by-default `.git/info/exclude` block, and seeded `.tu-agent` config.
2. If the concept store is empty, runs the `learn` pipeline (concept index +
   definitions + architecture synthesis) so the dev-flow has knowledge to draw on.

The dev-flow agents are **not** materialized as `.claude/agents/*.md`. The tdd
flow composes each role at runtime from an embedded generic shell (the role's
base contract: tools, output format, definition-of-done), the runtime language
overlay, your project rules, and the graph's conventions (`get_context`). There
is nothing to enrich. To customize a role, drop your own
`.claude/agents/<role>.md` into the repo â€” it wins over the embedded shell.

You can also run the pieces separately: `tu-agent prepare` (CLI) for the
deterministic setup only, or `/tu-agent:learn` to (re)build the knowledge index.
Commit `settings.json` with the rest of your team.

## Project rules

`.tu-agent/rules.md` is a **user-owned** file for repo-specific rules the dev-flow
must follow (conventions the generated agent bodies do not already encode). An
optional `.tu-agent/rules/<role>.md` (roles: `analyst`, `architect`, `developer`,
`pr-reviewer`, `scribe`) adds rules for one role only.

These files are yours: **no tu-agent command ever writes them**, so hand edits
survive every `prepare` and `--update`. Both are optional â€” absent or blank files
change nothing.

When present, the rules are spliced into each TDD stage prompt between the agent
body and the stage overlay (`body + rules + overlay`), so every dispatched stage
sees them. The judge enforces them: a change that violates a project rule is
grounds to `revise`, citing the exact rule. (The judge criterion lives in the
binary, so it reaches the plugin with the next `tu-agent` release.)

## Compatibility with the binary

Knowledge generated by the plugin is identical to that generated by
`tu-agent learn` â€” both persist one thin concept card per concept into the graph
store (`.tu-agent/graph.db`), read via `get_concept`. The architecture overview
is likewise stored in `graph.db` (metadata), read via `get_architecture` â€” no
`.claude/skills` file. The concept index is reproducible: re-running either path
regenerates the same cards deterministically.
