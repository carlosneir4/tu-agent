# tu-agent Claude Code plugin

Runs the tu-agent knowledge pipeline inside Claude Code, on your Claude
subscription — no API key needed. The tu-agent binary does everything
deterministic (graph build, concept clustering, card rendering, CLAUDE.md
registration); Claude Code does everything generative (concept definitions,
architecture overview).

## Requirements

- The `tu-agent` binary, version 0.3.0+, on your `PATH` or at
  `~/.tu-agent/bin/tu-agent`:

  ```bash
  go install -tags sqlite_fts5 github.com/tu/tu-agent/cmd/tu-agent@latest
  ```

## Install

From this repository:

```bash
claude --plugin-dir /path/to/tu-agent/plugin
```

## Commands

| Command | What it does |
|---|---|
| `/tu-agent:prepare [path]` | End-to-end repo setup: binary init (agents, CLAUDE.md, hardened settings.json) → learn pipeline (if needed) → per-role agent enrichment |
| `/tu-agent:learn [path]` | Full pipeline: graph build → concept index (deterministic) → one-line definitions (in-session) → architecture synthesis → CLAUDE.md registration |
| `/tu-agent:synthesize` | Regenerate the architecture skill from the existing concept index |
| `/tu-agent:status` | Progress, skill staleness, and graph health |

## Graph MCP server

Installing this plugin registers a stdio MCP server, `tu-agent-graph`, that
Claude Code starts automatically. It runs the bundled binary
(`${CLAUDE_PLUGIN_ROOT}/bin/tu-agent mcp`) against the project you opened Claude
Code in, exposing graph query tools and test-generation helpers:

| Tool | Surfaces as | What it answers |
|------|-------------|-----------------|
| `get_impact` | `mcp__tu-agent-graph__get_impact` | Blast radius: what breaks if I change this symbol? |
| `get_context` | `mcp__tu-agent-graph__get_context` | Everything relevant to touching a target: blast radius, skills, conventions, tests. |
| `find_symbol` | `mcp__tu-agent-graph__find_symbol` | Where is this symbol defined? |
| `test_gaps` | `mcp__tu-agent-graph__test_gaps` | Rank untested public functions by risk (fan-in × blast radius). |
| `test_scaffold` | `mcp__tu-agent-graph__test_scaffold` | Deterministic half of test generation: context, test path, scoped run command. |
| `get_concept` | `mcp__tu-agent-graph__get_concept` | Return a concept card (SKILL.md) by name, or list all concepts. |
| `get_traits` | `mcp__tu-agent-graph__get_traits` | Trait view: shared interfaces + logic sites (as_type) and implementers + blast radius (as_interface). JSON. |
| `get_flow` | `mcp__tu-agent-graph__get_flow` | Execution call tree from an entry symbol: boundary crossings, interface dispatch candidates, depth + fan-out control. JSON. |

The tools read `./.tu-agent/graph.db`. Build it first with `/tu-agent:learn`
(or `tu-agent graph build`); until a graph exists the queries return empty
results. The server refreshes the graph from the working tree each time it
starts.

## Plugin skills

| Skill | What it does |
|-------|--------------|
| `init` | End-to-end repo setup — binary init, learn pipeline (if needed), per-role agent enrichment. |
| `learn` | Full learn pipeline — concept index, definitions, architecture synthesis, CLAUDE.md registration. |
| `synthesize` | Regenerate the architecture skill from existing concept cards. |
| `status` | Skill staleness and graph health. |
| `test-gen` | Generate one verified unit test for a target symbol. Graph context comes from MCP; test body is generated by Claude Code. |

## Dev-flow agents

`/tu-agent:prepare` sets up the repo end-to-end:

1. Runs the binary's deterministic setup — five skeleton dev-flow agents
   (`.claude/agents/`), a `CLAUDE.md`, a hardened `.claude/settings.json`
   (deny-wins permissions, a secret-guard hook, a toolchain formatter hook, and
   an `enabledMcpjsonServers` allowlist), and a tu-agent block in `.gitignore`.
2. If the concept store is empty, runs the `learn` pipeline (concept index +
   definitions + architecture synthesis) so the agents have knowledge to draw on.
3. Dispatches the `agent-enricher` agent per role, replacing each bare CLI
   skeleton with the curated graph-first agent (graph-first protocol and graph
   MCP tools come baked into that skeleton) and filling its `## Project Context`
   from the concept cards.

You can also run the pieces separately: `tu-agent prepare` (CLI) for the
deterministic setup only, or `/tu-agent:learn` to (re)build the knowledge index.

The base contract of each agent (tools, output format, definition-of-done) comes
from the versioned skeleton; the `<!-- ENRICH: -->` slots and `__PROJECT__`
tokens are filled per project. Commit the enriched agents and `settings.json`
with the rest of your team.

## Compatibility with the binary

Knowledge generated by the plugin is identical to that generated by
`tu-agent learn` — both persist one thin concept card per concept into the graph
store (`.tu-agent/graph.db`), read via `get_concept`. Only the `architecture`
skill is written as a file. The concept index is reproducible: re-running either
path regenerates the same cards deterministically.
