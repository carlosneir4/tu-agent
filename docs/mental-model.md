# Mental model: three organs of project knowledge

Read this once. Everything else in tu-agent follows from it.

The old way to help an AI understand a codebase was to generate prose docs per
module. Prose rots: the code changes, the doc doesn't, and now the assistant is
confidently wrong. tu-agent replaces that with **three complementary stores**,
each good at a different kind of question.

| Organ | What it is | Always accurate? | Answers |
|-------|-----------|------------------|---------|
| **Graph** | The live code structure — symbols and the `calls` / `implements` / `extends` / `imports` / `overrides` edges between them, in SQLite. Rebuilt incrementally from a diff. | **Yes** — derived from source | *What breaks if I change X? Who calls this? Where is it defined? What's the call flow? Which symbol is a chokepoint?* |
| **Concept index** | One thin **card** per concept (usually a package): a vocabulary → landmarks map. ≤1 KB each, stored in `graph.db`. | Structurally yes; the one-line *definition* is generated | *What does this part of the system mean? What are its landmark types? Which concepts exist?* |
| **Memory** | Durable, topic-keyed facts with provenance and revisions, in SQLite. Survives across sessions; links to graph nodes. | It's **curated truth**, not derived | *What did we decide / learn here? What's fragile? What relates to this node?* |

The split is deliberate:

- **Structure questions → the graph.** Exact, cheap, no model call.
- **Meaning questions → the thin cards.** "What is this area about?"
- **Hard-won facts → memory.** The *why* the code can't show you.

An agent loads a compact index up front and pulls detail only when it needs it —
instead of re-reading dozens of files to reconstruct what the graph already knows.

---

## 1. The graph — structure, exact and cheap

The graph is the backbone. It stores your symbols (functions, methods, types)
and the real edges between them, extracted from source. Because it's derived from
the code, it's never out of date after a rebuild — and rebuilds are incremental,
driven by a SHA-256 diff of what changed.

**Why it matters:** the expensive question in AI-assisted coding is *"what breaks
if I touch this?"* Without a graph, the assistant re-reads files, greps for
callers, and still misses framework or dependency-injection wiring. With the
graph, that's one cheap query that returns the actual dependents.

**Concrete example.** You want to change `payment.Gateway.Authorize` in the
`github.com/acme/orders` service. `get_impact` returns:

```
Symbol:  payment.Gateway.Authorize
Direct dependents (4):
  - orders.Service.Charge          orders/service.go:88
  - orders.Service.Refund          orders/service.go:140
  - billing.Reconciler.Settle      billing/reconcile.go:52
  - orders_test.TestCharge_Declined orders/service_test.go:210
Blast radius: 11 symbols across 3 packages
Tests to run: ./orders/... ./billing/...
```

That's the difference between "let me read the codebase" and "here's the answer."

> **The graph can miss framework/DI/compiled relationships.** If it returns
> "(none)" for a symbol you can see used, cross-check with a targeted search.
> tu-agent's own protocol tells the agent to do exactly this.

---

## 2. The concept index — meaning, thin and near-static

A **concept** is usually a package or a cohesive area. Its **card** is a small
map: the vocabulary of that area (the words you'd use to talk about it) pointing
to the **landmarks** (the key types/functions that anchor it), plus a
one-line definition.

Cards are deliberately thin — ≤1 KB. They are not documentation you read
top-to-bottom; they are a **routing table**. "The user is asking about billing →
the `billing` concept's landmarks are `InvoiceService`, `Reconciler`,
`LedgerEntry` → start there."

The structural part (vocabulary, landmarks) is derived deterministically by the
binary and stays accurate. Only the human-readable one-line definition is
generated by a model — and it's cheap to regenerate.

Cards are clustered automatically. On a large repo the binary uses community
detection (Leiden) and, when a cluster is too big, recursive sub-clustering, so
you get concepts at a useful granularity instead of one giant blob.

---

## 3. Memory — the *why*, curated and durable

The graph and cards are derived from code. **Memory is not** — it's the facts a
human decided are worth keeping: a design decision and the alternatives rejected,
a bug pattern and how to prevent it, a gotcha (a trap that bit someone), a
pointer to an external resource.

Each note has:

- a **topic key** like `decision/plugin-first` or `gotcha/stale-plugin-hooks`
- a **type** — one of `architecture`, `bug-pattern`, `decision`, `gotcha`,
  `reference`, `testing`
- **provenance and revisions** — who wrote it, when, what it superseded
- optional **links to graph nodes**, so "what do we know about `InvoiceService`?"
  can surface the relevant notes

**Why a separate store:** decisions and hard-won lessons evaporate session to
session and dev to dev. Memory keeps them, and — crucially — **shares them across
the team through git** (see [Memory](memory.md)). The next person to touch
`Reconciler` sees the note that explains why it retries three times.

---

## How the three work together

A realistic slice of a session:

1. You ask to add a feature to billing. The agent runs `get_context` on the
   billing area → gets the **concept** (what billing means), its **landmarks**,
   the **blast radius** of the entry point, and the **memory notes** attached to
   those nodes — all at once, all cheap.
2. It reads only the two files that actually matter, not the whole package.
3. It makes the change, and the graph **updates incrementally** on save.
4. It records the non-obvious decision it made as a **memory note**, which is
   **exported to a chunk** at session end for the rest of the team.

Structure, meaning, and history — each answered by the store that's best at it.
